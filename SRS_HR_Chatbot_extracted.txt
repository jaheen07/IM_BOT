This document covers the architectural design, functional requirements, and external interfaces for the backend system, which includes a FastAPI-based server and a Retrieval-Augmented Generation (RAG) engine. The intended audience for this document includes development teams, project managers, and stakeholders involved in the deployment of the HR assistance system.Document ConventionsAll mandatory requirements use “shall”; optional or recommended behaviors use “should.”Technical terms such as RAG, FAISS, embedding model, session context, LLM follow standard AI engineering terminology and are defined in the Glossary.Code elements mentioned in this document follow Python naming conventions exactly as in the implementation.Intended Audience and Reading SuggestionsThis document is intended for:AI Developers & ML Engineers To understand the architecture, data pipeline, and system integration points.Project Managers To track requirement completion and development progress.QA/Test Engineers To derive test cases from functional and non-functional requirements.DevOps / Infrastructure Engineers To understand deployment constraints, API endpoints, and runtime behaviors.Reading Suggestion:Developers should focus on Section 3 (System Features) and Section 4 (External Interface Requirements). Project Managers should prioritize Section 1.4 (Project Scope) and Section 2 (Overall Description).Project ScopeThe ACME AI HR Chatbot is a specialized software system designed to automate Human Resources inquiries. It utilizes a Large Language Model (LLM), specifically “meta-llama/Llama-3.3-70B-Instruct”, to answer user questions based on uploaded company data.Key capabilities include:Data Ingestion: The system processes company policies (PDF text) and employee details (tabular data), converting them into searchable vector embeddings using Sentence-Transformer and FAISS.Hybrid Retrieval: It employs a strategy that distinguishes between unstructured text (policies) and structured rows (employee CSV data) to ensure accurate retrieval.Contextual Awareness: The system resolves pronouns (e.g., "his", "her") by tracking the "current employee" discussed in the active session.Multi-User Session Management: The backend isolates user contexts via UserSession objects, ensuring that one user's conversation history does not bleed into another's, with automatic cleanup for sessions inactive for over one hour.Reinforcement Learning: The system tracks query patterns and accepts user feedback ratings (1-5) to refine future responses.The goal of this software is to provide a live, API-accessible backend that allows employees to query HR data seamlessly via natural language.ReferencesThe following documents and resources are referenced:Project Source Code app.py – Code file for API and session manager chatbot.py – Base code for RAG engine, embedding pipeline, LLM integrationHugging-Face Inference API Documentation URL: huggingface.co/docs/api-inferenceSentence-Transformers Model Documentation Model: all-MiniLM-L6-v2 URL: FAISS Library Documentation URL: github.com/facebookresearch/faissLlama 3.3 70B Model Card (SambaNova Hosted) URL: huggingface.co/meta-llamaOverall DescriptionProduct PerspectiveThe HR Chatbot is a backend software system designed to function as an intelligent intermediary between company employees and internal Human Resources management workframes. It operates as a self-contained API service built on the FastAPI framework, which exposes RESTful endpoints to potential frontend applications. The system is not a standalone desktop application but rather a server-side component that integrates with external services, specifically the Large Language Model (LLM) capabilities. It manages its own internal state for user sessions and data indexing but relies on a pre-defined PDF document (eg.; company_policies.pdf) as its primary knowledge source.Product FeaturesThe system provides a comprehensive set of features designed to facilitate natural language interaction with structured and unstructured HR data.Retrieval-Augmented Generation (RAG): The core feature is the ability to ingest a company PDF, extract text and tables, and index them using FAISS for semantic retrieval. This allows the system to ground its answers in specific policy documents or employee records.Multi-User Session Management: The application supports concurrent users by isolating conversation contexts into distinct User-Session objects. This ensures that variables like "current employee" or "chat history" are unique to each user ID.Contextual Understanding: The chatbot actively resolves pronouns (e.g., "his", "her") by tracking entities mentioned in the active conversation session, ensuring fluid follow-up questions are understood correctly.Reinforcement Learning and Analytics: The system includes a response re-learning loop where user previous query patterns are tracked to improve future performance.Administrative Monitoring: The API includes endpoints for server status checks, session monitoring (/api/sessions), and manual session cleanup to manage server resources effectively.User Classes and CharacteristicsThe project identifies two primary user classes and one developer classes:General Employees (End Users): This class represents the primary user base. They interact with the system via the “/api/chat” endpoint to ask questions about company policies or employee details. They require no technical expertise and expect natural language responses. They just input their queries to chatbot to get their desired answer.System Administrators: This class interacts with maintenance endpoints. They are responsible for monitoring system health via “/api/health”, managing active sessions via “/api/sessions”, and triggering manual cleanup of expired sessions.Backend Developers: These users interact with the raw code and configuration, managing environment variables like HF_TOKEN and ensuring the PDF data source is correctly formatted and located.Operating EnvironmentThe system is designed to operate in the following environment:Backend ServerPython 3.10+FastAPIUvicorn (host: 0.0.0.0 and port: 7860)Running on Linux (Ubuntu recommended)External DependenciesHugging-Face Router API (internet required)FAISS CPU versionSentenceTransformers modelsPyPDF2 & Tabula for PDF extractionFront-end ClientsAny web/mobile interface sending JSON requestsMust support CORSHardwareRecommended: 4–8 vCPUs, 8–16 GB RAMFAISS and embeddings processed locally; LLM inference outsourced to HF APIRequired Packages:The system requires the below python packages for setup the codebase environment:fastapi==0.104.1 uvicorn[standard]==0.24.0 pydantic==2.5.0 PyPDF2==3.0.1 faiss-cpu==1.7.4 numpy==1.24.3 sentence-transformers==2.2.2 huggingface-hub==0.19.4 pandas==2.0.3 tabula-py==2.9.0 python-multipart==0.0.6Design and Implementation ConstraintsAs an API-first product, user documentation is provided dynamically through the FastAPI automatic documentation interfaces. Upon launching the server, interactive documentation is available at the /docs (Swagger UI) and /redoc endpoints. These interfaces describe the request schemas (e.g., ChatRequest), response models, and available HTTP methods for developers integrating with the API.External API Dependency: The system's intelligence is strictly bound to the availability and performance of the Hugging Face Inference API (meta-llama/Llama-3.3-70B-Instruct).Session Concurrency: To maintain performance, the system enforces a hard limit of 100 concurrent active sessions. A threading lock is implemented to ensure thread safety during session creation and deletion.Session Timeout: User sessions are ephemeral and are automatically marked for deletion if inactive for 3600 seconds (1 hour).Data Format: The system is constrained to process input data from a single PDF file located at a specific path (default: ./data/policies.pdf), and table extraction relies on the specific formatting detectable by Tabula.Dependencies: The system depends on the HF_TOKEN environment variable being set for authentication with Hugging Face. It also depends on the sentence-transformers/all-MiniLM-L6-v2 model for local embedding generation.Assumptions: It is assumed that the uploaded PDF contains readable text and properly formatted tables that tabula-py can parse. It is also assumed that the server has write access to the local ./ directory to save history files (chat_history.pkl) and chunk manifests.System FeaturesThe following sections describe the major functional capabilities of the ACME AI HR Chatbot backend. Each feature includes its purpose, interaction flow, and detailed requirements.Multi-User Session Management3.1.1Description and PriorityPriority: High This feature allows the system to handle multiple users simultaneously by creating isolated session contexts for each unique user ID. It ensures that conversation history and contextual entities (e.g., "current employee being discussed") are maintained separately for every user, preventing data leakage between concurrent conversations.3.1.2Stimulus/Response SequencesStimulus: A client sends a request to /api/chat or /api/history with a specific user_id.Response: The system checks the user_sessions dictionary. If the ID exists, it retrieves the session. If not, it creates a new UserSession object and logs the creation.Stimulus: A session remains inactive for longer than the configured timeout (3600 seconds).Response: The system automatically removes the session data during the next cleanup cycle to free up memory.3.1.3Functional RequirementsREQ-3.1.1: The system shall maintain a global dictionary to store active user sessions, protected by a threading lock to ensure thread safety during concurrent access.REQ-3.1.2: The system shall limit the maximum number of active sessions to 100 (MAX_SESSIONS) to prevent memory overflow.REQ-3.1.3: The system shall automatically identify and delete sessions that have been inactive for more than 3600 seconds (1 hour).REQ-3.1.4: The system must provide a manual endpoint /api/reset to immediately clear the chat history and context for a specific user ID upon request.Retrieval Augmented Generation (RAG) Engine3.2.1 Description and PriorityPriority: High The RAG engine is the core intelligence module that processes theuploaded PDF document (policies.pdf). It is responsible for extracting text, parsing tables into structured data, generating vector embeddings, and retrieving relevant information based on user queries.3.2.2 Stimulus/Response SequencesStimulus: The application starts up (Initialization phase).Response: The system loads the PDF, extracts tables into a CSV, chunks the text, builds a FAISS index, and initializes the embedding model.Stimulus: The RAG Chatbot receives a resolved query.Response: The system searches the vector index for the top 20 most relevant chunks (k=20) and returns them as context for the LLM.3.2.3 Functional RequirementsREQ-3.2.1: The system shall use PyPDF2 to extract unstructured text and tabula-py to extract structured tables from the source PDF.REQ-3.2.2: The system shall implement a hybrid chunking strategy that differentiates between "Text Chunks" (policy sections) and "Table Chunks" (individual employee rows) to optimize retrieval accuracy.REQ-3.2.3: The system shall generate vector embeddings for all data chunks using the sentence-transformers/all-MiniLM-L6-v2 model.REQ-3.2.4: The system shall utilize a FAISS (Facebook AI Similarity Search) index to perform efficient similarity searches on the generated embeddings.Context-Aware Chat Interface3.3.1 Description and PriorityPriority: High This feature manages the conversational flow. It processes user input,resolves linguistic references (pronouns), constructs a prompt enriched with retrieved data and chat history, and communicates with the external LLM API to generate a response.3.3.2 Stimulus/Response SequencesStimulus: User inputs a question containing pronouns (e.g., "What is his email?").Response: The system analyzes the session context to identify the current_employee and rewrites the query (e.g., "What is John Doe's email?") before processing.Stimulus: The system has gathered all necessary context (retrieved chunks + chat history).Response: The system sends a POST request to the Hugging Face Router API with the constructed prompt and awaits the generated text response.3.3.3 Functional RequirementsREQ-3.3.1: The system shall expose a POST endpoint /api/chat that accepts a JSON payload containing question and user_id.REQ-3.3.2: The system must implement context resolution logic to replace pronouns (he, she, his, her, they) with the name of the entity currently being discussed in the user's session.REQ-3.3.3: The system shall construct a prompt that includes the User Query, Relevant PDF Chunks, and the last 5 relevant past conversation turns to maintain conversational continuity.REQ-3.3.4: The system shall utilize the meta-llama/Llama-3.3-70B-Instruct model via external API to generate natural language responses.REQ-3.3.5: The system shall enforce a temperature setting of 0.3 and a max token limit of 512 for LLM generation to ensure concise and factual answers.System Monitoring and Maintenance3.4.1 Description and PriorityPriority: Medium This set of features provides endpoints for system administrators to monitor the health of the application, view active session statistics, and manually trigger maintenance tasks.3.4.2 Stimulus/Response SequencesStimulus: Administrator requests system status.Response: The system returns a JSON status report including the "healthy" state, chatbot initialization status, and count of active sessions.3.4.3 Functional RequirementsREQ-3.4.1: The system shall provide a GET endpoint “/api/health” that returns HTTP 200 OK only if the base_chatbot has been successfully initialized.REQ-3.4.2: The system shall provide a GET endpoint “/api/sessions” that lists all active user IDs, their message counts, and last activity timestamps.REQ-3.4.3: The system shall provide a POST endpoint “/api/cleanup” to allow administrators to manually trigger the removal of expired sessions.External Interface RequirementsUser InterfacesThe system functions as a configuration.Other Nonfunctional RequirementsPerformance RequirementsPERF-01: The system shall retrieve and embed a user query within ≤ 300 ms on average under standard server load.PERF-02: FAISS semantic search for relevant chunks shall execute in ≤ 100 ms for top-k retrieval (k ≤ 20).PERF-03: End-to-end response time (from /api/chat request to answer) shall be ≤ 2.5 seconds, excluding external LLM inference time.PERF-04: The system shall handle at least 50 concurrent users without degradation of response quality or timeout errors.PERF-05: Session creation and lookup shall complete within ≤ 50 ms using the in-memory session store.PERF-06: The system shall process and index a typical HR policy PDF (up to 200 pages with tables) within 2–5 minutes during ingestion.PERF-07: The system shall maintain FAISS search accuracy above 90% relevance match for employee and policy queries.Safety RequirementsSAFE-01: The system shall prevent disclosure of employee information beyond what the user explicitly asks for (e.g., shall not reveal full employee profiles unnecessarily).SAFE-02: If retrieved context is incomplete or ambiguous, the system shall respond with:“I don’t have this information.”SAFE-03: In case of LLM failure, timeout, or API error, the system shall return a safe fallback message without crashing.SAFE-04: The system shall prevent the LLM from hallucinating nonexistent employees by strictly grounding responses in retrieved table chunks.SAFE-05: If the uploaded PDF contains corrupted or unreadable data, the system shall abort ingestion safely and notify administrators.Security RequirementsSEC-01: The system shall ensure that user sessions are isolated and no user can access another user’s history or context.SEC-02: All communication between backend services and HuggingFace API shall use HTTPS with TLS encryption.SEC-03: The system shall not store or log API tokens (HF_TOKEN) in plaintext logs.SEC-04: Employee-related data extracted from the PDF shall remain on the server and shall not be transmitted to third parties except where necessary for query context passed to the LLM.SEC-05: The system shall sanitize all user inputs to prevent prompt injection as much as possible.SEC-06: Sensitive files (chat history, embeddings, table CSV, stats) shall be stored with restricted filesystem permissions.SEC-07: The system shall enforce session timeout (1 hour) to protect stale session data from being exposed.SEC-08: If the system detects more than 100 active sessions (MAX_SESSIONS), it shall automatically clean up inactive sessions to prevent malicious flooding.Software Quality AttributesUsabilityQUAL-USE-01: The system shall produce structured, readable answers without unnecessary verbosity.QUAL-USE-02: Responses shall be deterministic and consistent for similar queries across users.MaintainabilityQUAL-MAIN-01: The backend codebase shall follow modular structure (RAG engine, API, session handler, utils).QUAL-MAIN-02: Developers shall be able to add new features (e.g., new embedding model) with minimal changes to core modules.ScalabilityQUAL-SCALE-01: The system shall support scaling horizontally by running multiple instances behind a load balancer.TestabilityQUAL-TEST-01: Each module (retrieval, chunking, LLM handler, session manager) shall support isolated unit testing.QUAL-TEST-02: API endpoints shall
